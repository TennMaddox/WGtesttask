Task №1:
Для работы с БД создаю отдельную директорию db, в которой будут располагаться все схемы и методы для работы с базой.
Так как у нас одна схема, создаю модуль в котором описываю структуру базы, там же создаем функцию по созданию этих таблиц в блоке try except, чтобы в случае ошибок не блокировали соединение. В функции используем executescript, чтобы сразу несколько SQL запросов за один вызов выполнить.
Для реального проекта создание подобных БД имело бы смысл разделить на раздельные миграции, но в тестовом будем придерживаться принципа YAGNI – you ain`t gonna need it помимо основных паттернов.
Task № 2:
Создаем модуль create_and_fill в той же директории, основными моментами реализации являются – две обозначенные константы для соблюдения условия по рейнджу целочисленных данных от 1-20, вспомогательный метод _rand (приватный) для уменьшения дупликации кода и сам метод, который последовательно заполняет случайную инфу по модулям и кораблям. Метод так же обернут в обработчик исключений, чтобы блок finally закрыл соединение даже в случае получения ошибки.
Task № 3:
В целом, ради объяснения имплементации этого задания и создается этот файл.
Так как тесты не должны тестировать БД напрямую, то нам требуется создать классы для каждого компонента корабля и самого корабля. Создаю директорию app, в которой будут описаны классы. Так как наши классы не должны иметь внутри себя никаких методов и не выполняют никаких действий, то по сути они идеально подходят под описание dataclass. Что означает что мы создадим модель каждого класса и репозиторий компонента. 
Отсюда, в директории арр создаем две новые папки – models и repositories. В папке models создаем отдельный модуль для каждого компонента и отдельно для корабля. В итоге получаем 4 модели. Для каждой модели создается свой репозиторий, он содержит в себе:
1. Конструктор, который принимает соединение с БД
2. Импорты своей модели и другие
3. Метод get_by_name – выполняет запрос в бд с параметром, который передается и устанавливается в плэйсхолдер в запросе, после чего происходит распаковка кортежа, чтобы передать полученные данные в аргументы конструктора соответствующей модели. Этот же объект и возвращает метод. 
Репозиторий представляет одну таблицу как интерфейс.

Далее, самый сложный компонент задания – рандомайзер БД. 
Начнем с более детального рассмотрения условия задания – рандомайзер должен во-первых менять один модуль каждому кораблю на случайный, а во-вторых менять каждому компоненту один случайный атрибут. Отсюда следует, что все наши тесты из будущего задания 4 должны проваливаться, ведь у нас по сравнению с оригинальной БД всегда будет либо другой компонент у корабля (только если не пришел тот же самый или не поменялся не тот компонент, что проверяется на текущий момент (поменяли оружие в тесте двигателей)), либо упадет потому что будет разница в атрибутах компонента (у weapon1 изменили rotational_speed). Возможный сценарий для прохождения тестов – если значение атрибута будет изменено на то же значение, что и было. Это не противоречит условию задания, поэтому оставляем этот момент.
Также, надо предусмотреть, что после того, как кораблю поменяли компонент, мы должны обновлять атрибут у уже обновленного компонента (был weapon1, поменяли на weapon2 – обновляем атрибут weapon2). Помимо этого, если однажды мы обновили атрибут компоненту, при дальнейшем его изменении мы накопим n-количество изменений. Тогда возможна ситуация, что тест№1 упадет, потому что у weapon1 поменяли 1 атрибут, а тест№200 упадет, потому что у weapon1 изменены уже все атрибуты – это видится неправильным, значит избегаем накапливания изменений. Однажды измененный компонент не должен обновлять атрибуты более одного раза, передаем всем кораблям ссылки на уже измененные компоненты.
Теперь, по порядку о самой реализации. 
0. Конструктор, в котором сохраняем пути к оригинальной БД и к рандомной.
1. Метод create_randomized_copy создает полную копию оригинальной БД, открывает с ней соединение, внутренним методом класса _randomize мутируем данные, фиксируем данные и закрываем соединение.
2. _rand_param – такой же метод по генерации рандомного инта в указанных условием задания пределах.
3. Метод _load_data используется для загрузки данных из БД, чтобы вычитать списки всех кораблей и компонентов. Вычитка также происходит с использованием вспомогательных внутренних методов класса, чтобы избежать массивной дупликации кода (_get_ships_from_table / _get_component_list_from_table).
4. _get_component_list_from_table получает на вход таблицу и название компонента, чтобы потом выполнить селект по этим данным и преобразовать эти данные в список компонентов
5. Вспомогательный метод для смены компонента кораблю – update_ship_component – вынесен в отдельный метод для уменьшения дупликации. Сам метод представляет собой скрипт для обновления БД, который принимает изменяемый компонент, новое значение (новый компонент) и номер корабля, которому передаем изменения.
6. Теперь вспомогательный метод для смены параметра у компонента _update_component_param_once. Метод сначала проверяет, не является ли компонент для изменения уже измененным. Далее мы выбираем случайное поле для изменения и его новое значение. Далее идет блок кода для логирования (был добавлен в процессе отладки, так как не сразу удалось соблюсти всю последовательность логики и требования. Решил его оставить, потому что так легче проверять результаты и видеть, как произошли изменения, что с чем сравниваем. Для упрощения, далее не буду описывать моменты, где мы что-то логируем). Далее, обновляем значение параметра и добавляем измененный модуль в список измененных.
7. Метод _change_component – представляет собой основной модуль логики по изменению компонента кораблю. Сначала рандомно выбирается компонент, который будет подвергнут замене. Далее, однотипная логика кода, где в зависимости от типа выбранного компонента мы выбираем новый компонент, логируем инфу и вызываем вспомогательный метод для апдейта в БД, так же возвращаем все имеющиеся компоненты. Сначала думал использовать конструкцию match-case, но прочитал, что If работает быстрее при количестве условий до трех, поэтому оставил так.
8. Метод _randomize – Основной метод рандомизации, который строит свою логику на использовании вспомогательных методов. Он уже учитывает все требования, описанные выше. Сначала создаем три сета, в котором будем хранить измененные компоненты, и выполняем условие задания пункта А – меняем рандомный компонент кораблю на другой рандомный. После чего, последовательно выполняем условие задания пункта В – меняем по случайному атрибуту каждого компонента – начинаем с оружия, заканчиваем двигателем.

Таким образом, получили весьма сложный, но последовательный и декомпонизированный рандомайзер БД, который удовлетворяет исходной постановке задания.
Далее, выводим запуск этих методов в фикстуру.
Тесты и conftest будем создавать в отдельной директории tests.
Первая фикстура создает оригинальную БД, вызывает метод по заполнению БД и возвращает путь к готовой БД, скоуп указываем согласно заданию Session, чтобы БД создавалась один раз за всю сессию тестов и оригинальная БД не менялась.
Вторая фикстура представляет собой почти то же самое, только в ней мы создаем экземпляр класса DatabaseRandomizer и вызываем у него метод по созданию рандомихированной БД, также возвращаем путь.
Третья фикстура original_conn нужна для открытие соединения с оригинальной БД и передачи ее тестам. 
Тоже самое делаем и с четвертой фикстурой, но для рандомизированной БД.
Далее пятая фикстура, в которой мы создаем и возвращаем 8 репозиториев – по 1 на каждый объект оригинальной БД и по 1 на каждый объект Рандомизированной БД. Это нужно так как тесты не напрямую смотрят в базу и сравнивают оттуда значения, а потому что они будут работать с объектами моделей, а не со строками из БД.
